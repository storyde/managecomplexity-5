Internationalisation Plan

Goal

- Keep one shared `game.js`.
- Translate only player-visible UI text in the frontend.
- Leave scene content and scene-source translation out of scope.
- Keep `displayParagraphHTML()` mostly unchanged.
- Keep the existing English speaker-prefix mapping in scene output, but localize the visible speaker label in the chat bubble.

Scope

- `out/html/game.js`
- `out/html/index.html`
- A small shared i18n dictionary file or inline dictionary block

Why `index.html` should be included

- A large part of the visible UI is hardcoded in `index.html`, not `game.js`.
- If only `game.js` is internationalized, the player would still see English in:
  - header button labels and titles
  - sidebar headings
  - modal title and table headings
  - footer navigation buttons
  - ARIA labels for major UI regions
- So it makes sense to treat `index.html` and `game.js` as one player-visible UI layer.

Recommended approach

- Add a small translation dictionary for UI text only.
- Use `document.documentElement.lang` as the active language.
- Add a simple `t(key, vars)` helper with English fallback.
- Replace hardcoded JS strings with translation keys.
- Mark translatable HTML nodes in `index.html` with `data-i18n` and `data-i18n-attr`.
- Apply translations on page load from one place.

Suggested structure

- `out/html/game.js`
  - UI logic
  - `t()` helper if kept inline
  - `applyStaticTranslations()` if kept inline
- `out/html/index.html`

  - keep structure unchanged
- Optional new file:

```js
function getCurrentLang() {
  return document.documentElement.lang || 'en';
}

function t(key, vars = {}) {
  const lang = getCurrentLang();
  const dict = I18N[lang] || I18N.en;
  let text = dict[key] || I18N.en[key] || key;
  Object.keys(vars).forEach(function(name) {
    text = text.replace('{' + name + '}', vars[name]);
  });
  return text;
}
```

Plan for `displayParagraphHTML()`

- Keep the existing English prefix mapping exactly as it is.
- Do not change scene-source conventions.
- Only change the visible labels rendered in bubbles and `alt` text.
- Example:
  - keep parsing `Vendor: ` to get class `cve`
  - show `t('speakerVendor')` in the bubble instead of hardcoded `Vendor`
- This means the parser stays stable while player-visible labels become translatable.

What should change in `displayParagraphHTML()`

- Keep:
  - `mapping`
  - class names
  - profile image logic
  - prefix detection
- Change:
  - `characterNames` should become translation keys or translated values from `t()`
  - visible `Me` label in the self bubble should use `t('speakerMe')`
  - image `alt` text and sender labels should use translated speaker names

Plan for `index.html`

- Add translation hooks to visible labels instead of hardcoding text.
- Use one of these two patterns:

```html
<h3 data-i18n="sidebar.progress">Progress</h3>
```

```html
<button
  id="progress-toggle"
  data-i18n-attr="aria-label:header.toggleProgress,title:header.toggleProgressPanel"
  aria-label="Toggle Progress"
  title="Toggle Progress Panel">
</button>
```

- Then apply them once on page load:

```js
function applyStaticTranslations() {
  document.querySelectorAll('[data-i18n]').forEach(function(node) {
    node.textContent = t(node.getAttribute('data-i18n'));
  });

  document.querySelectorAll('[data-i18n-attr]').forEach(function(node) {
    node.getAttribute('data-i18n-attr').split(',').forEach(function(pair) {
      var parts = pair.split(':');
      var attr = parts[0];
      var key = parts[1];
      node.setAttribute(attr, t(key));
    });
  });
}
```

Implementation order

1. Add the i18n dictionary and `t()` helper.
2. Convert all player-visible strings in `game.js`.
3. Convert visible labels and ARIA labels in `index.html`.
4. Add a small `applyStaticTranslations()` pass for `index.html`.
5. Set `document.documentElement.lang` from the selected language before UI initialization.
6. Optionally switch save timestamps to `toLocaleString(getCurrentLang(), DateOptions)`.

What not to change

- Do not touch scene-file translation in this task.
- Do not change the English prefix mapping used by `displayParagraphHTML()`.
- Do not rebuild the chat rendering model unless needed later.
- Do not internationalize internal storage keys such as `TITLE`.

Low-risk first pass

- `game.js`
  - notifications
  - alerts
  - loading label
  - save/load button labels
  - ARIA labels created in JS
  - visible speaker names
- `index.html`
  - header labels
  - sidebar headings
  - footer button labels
  - modal headings and column headings
  - important ARIA labels

Expected result

- One shared frontend code path for all UI languages.
- No need to duplicate `game.js`.
- No need to change scene prefix mapping.
- Player-visible labels can switch per language cleanly.
- `index.html` and `game.js` stay aligned, so the UI does not end up half translated.
